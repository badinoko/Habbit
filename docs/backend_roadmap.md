# Backend Roadmap

## 1. Назначение документа
`backend_roadmap.md` переводит стратегию из `overview.md` в исполняемый план разработки backend-части HabitFlow.

Документ отвечает на вопросы:
- что именно нужно реализовать до финала диплома;
- в каком порядке это делать;
- какие критерии качества обязательны на каждом шаге;
- как проходит mentor-цикл (реализация + code review + фиксация выводов).

## 2. Целевой результат дипломной версии
К моменту финала должны быть полностью реализованы и продемонстрированы:
1. `Habits v1` end-to-end.
2. `Statistics v1` end-to-end.
3. `Auth v1` end-to-end.

Эти три направления имеют равный приоритет: ничего из списка не может быть отложено «на потом».
При этом обязательный слой реализации — текущий web-first интерфейс (`HTML + redirect + AJAX`), а отдельный `/api/*` остается дополнительным треком.

## 3. Что этот roadmap не покрывает
Этот roadmap:
- не является планом перехода на микросервисную архитектуру;
- не является планом оптимизации под высокую масштабируемость;
- не является roadmap по performance tuning;
- не является blueprint для production SaaS;
- не является обязательным планом построения полного API-first REST слоя.

Это учебный backend-roadmap для дипломного проекта.

## 4. Рамки и допущения
- Ты фокусируешься только на backend.
- Frontend реализуется агентом как поддерживающий слой для демонстрации backend-функций.
- Архитектурный стиль остается слоистым: `routers -> services -> repositories -> models/schemas`.
- В обязательный scope входят существующие web-роуты; `/api/*` проектируется как optional extension.
- Изменения делаются малыми итерациями, каждая итерация должна быть проверяемой тестами.

## 5. Quality Gates (обязательные)
Каждая backend-задача закрывается только при прохождении всех условий:
1. Полная типизация новой/измененной логики.
2. Тесты на ключевые позитивные и негативные сценарии.
3. Прохождение `ruff`, `pytest`, `mypy` (или релевантного поднабора).
4. Короткий code review с фиксацией:
   - найденных рисков;
   - пропущенных тестов;
   - технического долга и решения по нему.
5. Обновление документации при изменении контрактов/поведения.

## 6. Workstreams

### 6.1 Habits v1
Цель: полноценный backend-модуль привычек.

Скоуп v1:
- CRUD привычек.
- Отметка выполнения привычки по дате.
- Базовые правила периодичности (ежедневно/по дням недели).
- Текущий стрик (серия выполнений).

Артефакты:
- модели БД и миграции;
- репозиторий привычек и репозиторий событий выполнения;
- сервис с бизнес-правилами выполнения/пропуска;
- web-роуты и схемы для текущего UI-потока;
- unit + integration тесты;
- обновление документации контрактов;
- опционально: отдельные `/api/habits/*` эндпоинты.

### 6.2 Statistics v1
Цель: корректная backend-статистика без UI-зависимости.

Скоуп v1:
- агрегаты по задачам и привычкам;
- ежедневная/недельная динамика;
- метрики completion rate и streak summary;
- фильтры по диапазону дат.

Артефакты:
- сервис статистики (отдельный модуль);
- SQL/репозиторные агрегаты с проверяемой логикой;
- контракт выдачи статистики для web-слоя (dashboard/страницы);
- тесты на корректность подсчетов и edge cases (пустые данные, границы дат, timezone policy).
- опционально: отдельные `/api/statistics/*` эндпоинты.

### 6.3 Auth v1
Цель: базовая авторизация и защита изменяющих действий.

Скоуп v1:
- регистрация;
- логин/логаут;
- сессия пользователя;
- ограничение доступа к state-changing операциям;
- привязка данных к владельцу (минимум для привычек/задач).

Артефакты:
- user-модель и миграции;
- сервис аутентификации;
- зависимости/guards на текущих роутерах (в первую очередь state-changing);
- тесты на доступ, отказ, срок жизни сессии, security-базу;
- обновление env-конфигурации и README.

## 7. Порядок реализации (итерации)
Порядок выбран по зависимостям, а не по важности.

### Iteration 0: Stabilization Baseline
Цель: закрепить базу качества перед расширением функционала.

Задачи:
- выровнять тестовый контур;
- зафиксировать минимальный набор команд проверки;
- убрать известные расхождения между кодом/доками/конфигом.

Критерий выхода:
- стабильный запуск тестов локально и в контейнерном окружении.

### Iteration 1: Domain Contracts
Цель: описать контракты до активной реализации.

Задачи:
- зафиксировать web-first контракт текущих и новых роутов (`tasks/themes/habits/statistics/auth`);
- согласовать статусы ошибок и валидации;
- зафиксировать policy по времени (UTC/локаль) для статистики и streak.
- определить границы optional `/api/*` слоя без включения его в обязательный минимум.

Критерий выхода:
- утвержденные контракты + базовые failing tests/заглушки.

### Iteration 2: Habits v1
Цель: закрыть `habits` end-to-end.

Задачи:
- БД + сервис + роуты + тесты;
- интеграция в текущие dashboard-данные.

Критерий выхода:
- привычки полностью работоспособны и покрыты тестами.

### Iteration 3: Auth v1
Цель: закрыть базовую аутентификацию и защиту действий.

Задачи:
- регистрация, логин, логаут, сессии;
- защита state-changing роутов;
- user ownership для данных v1.

Критерий выхода:
- неавторизованный пользователь не может изменять данные.

### Iteration 4: Statistics v1
Цель: закрыть статистику как отдельный backend-модуль.

Задачи:
- агрегаты по задачам/привычкам;
- исторические метрики по датам;
- выдача статистики в web-слой.
- опционально: отдельный `/api/statistics/*` слой.

Критерий выхода:
- метрики корректны по тестовым сценариям и граничным случаям.

### Iteration 5: Final Hardening
Цель: довести проект до дипломной и портфельной готовности.

Задачи:
- ревизия безопасности `auth v1`;
- финальный прогон quality gates;
- финальная синхронизация документации;
- подготовка демонстрационного сценария.

Критерий выхода:
- проект проходит полный check-пайплайн и готов к защите.

## 8. Mentor Mode Workflow (на каждую задачу)
1. Формулируем цель задачи и минимальный scope.
2. Согласуем подход и риски.
3. Реализуем минимальный diff.
4. Пишем/обновляем тесты.
5. Запускаем проверки.
6. Проводим code review:
   - баги/регрессии;
   - архитектурные риски;
   - пробелы в тестах;
   - что улучшить в следующей итерации.
7. Фиксируем выводы в `engineering_journal.md` (и при необходимости кратко в `CHANGELOG.md`).

## 9. Checklist готовности к финалу
- [ ] `Habits v1` реализован end-to-end.
- [ ] `Statistics v1` реализован end-to-end.
- [ ] `Auth v1` реализован end-to-end.
- [ ] Type checking и тесты стабильны.
- [ ] Все state-changing операции защищены.
- [ ] Документация согласована с фактической реализацией.
- [ ] Подготовлен демонстрационный сценарий для диплома/собеседований.

## 10. Ближайшие шаги
1. Синхронизировать `api_contract.md` с web-first моделью и зафиксировать нормализации текущих роутов.
2. Начать реализацию нормализаций `N1..N3` и обновлять статусы в `api_contract.md`.
3. Начать Iteration 0 и закрыть baseline стабильности.
4. Вынести `/api/*` в отдельный optional backlog после обязательного минимума.
