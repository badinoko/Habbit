# Testing Strategy

## 1. Назначение документа
`testing_strategy.md` фиксирует тестовую стратегию HabitFlow для web-first разработки.

Цель:
- заранее определить, какие тесты обязательны для каждого типа изменений;
- сделать критерии готовности проверяемыми;
- избежать ситуации "код написан, но непонятно, что именно проверять".

## 2. Принципы тестирования
- Тесты проверяют поведение, а не внутреннюю реализацию.
- Каждый bugfix сопровождается регрессионным тестом.
- Для state-changing сценариев обязательны позитивный и негативный кейсы.
- Для web-first роутов проверяются и статус-коды, и формат ответа (`html/json/empty`).
- Изменения контрактов из `api_contract.md` считаются закрытыми только после тестов.

## 3. Уровни тестов

### 3.1 Unit (`tests/unit`)
Проверяют бизнес-логику сервисов в изоляции.

Примеры:
- валидация входных данных;
- маппинг приоритетов;
- правила уникальности/ограничений.

### 3.2 API-unit (`tests/api_unit`)
Проверяют контракт роутеров без поднятия реальной БД (через dependency overrides).

Примеры:
- статус-коды и response payload;
- edge-case ответы (`404`, `400`, `500`);
- smoke-проверки HTML страниц.

### 3.3 Integration (`tests/integration`)
Проверяют end-to-end поведение приложения с тестовой БД и миграциями.

Примеры:
- реальные CRUD-сценарии;
- корректность изменений в БД;
- интеграция роутер -> сервис -> репозиторий.

## 4. Матрица покрытия (текущий обязательный минимум)

| Область | Unit | API-unit | Integration |
|---|---|---|---|
| `themes` | обязательно | обязательно | обязательно |
| `tasks` | обязательно | обязательно | обязательно |
| `habits` | обязательно | обязательно | обязательно |
| `PUT /tasks/{id}` (JSON в success/error, `400/404/500`) | обязательно | обязательно | желательно |
| `GET /tasks/{id}` (`404` если задача отсутствует) | не требуется | обязательно | желательно |
| Content-Type контракт web/AJAX (`html/json/empty`) | не требуется | обязательно | обязательно |

Примечание:
- "желательно" становится "обязательно", если изменение затрагивает реальную БД/транзакционное поведение.

## 6. Набор обязательных кейсов на задачу
Для каждой backend-задачи минимум:
1. Happy path.
2. Один негативный бизнес-кейс.
3. Один edge-case (пустые данные/несуществующий ID/повторный вызов).
4. Проверка формата ответа для соответствующего типа роута.

## 7. Проверки перед завершением задачи
Минимальный локальный прогон:
1. `poetry run pytest -x tests -v` или `make test`
2. `poetry run ruff check src tests`
3. `poetry run mypy src --explicit-package-bases`

Полный прогон (при крупных изменениях):
1. `make check`

## 8. Правила добавления новых тестов
- Unit: `tests/unit/test_<service>_service.py`
- API-unit: `tests/api_unit/test_<router>_router_unit.py`
- Integration: `tests/integration/test_<domain>.py`
- Имена тестов: `test_<expected_behavior>_<condition>()`

## 9. Правило фиксации результатов
После закрытия задачи:
- в `api_contract.md` обновить фактический контракт, если изменились роуты/коды/формат ответов;
- в `backend_roadmap.md` обновить прогресс этапа;

## 10. Критерий готовности тестовой части
Тестовая часть задачи считается завершенной, когда:
- выполнена матрица покрытия для затронутой области;
- обязательные кейсы из раздела 6 присутствуют;
- проверки из раздела 7 пройдены;
- документация обновлена.
