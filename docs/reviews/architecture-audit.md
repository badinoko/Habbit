## 2. Архитектура приложения

### 2.1 Слоистая структура (Layered Architecture)

Проект следует классической 4-слойной архитектуре:

```
Routers (routers/)           → HTTP-обработка, валидация, рендеринг шаблонов
    ↓
Services (services/)         → Бизнес-логика, оркестрация
    ↓
Repositories (repositories/) → Доступ к данным, построение запросов
    ↓
Models (models/)             → SQLAlchemy ORM-определения
```

Это правильный подход: каждый слой имеет свою зону ответственности, что упрощает тестирование и поддержку.

**Сильные стороны:**
- Нет SQL-запросов в роутерах — вся работа с БД инкапсулирована в репозиториях
- Сервисы не знают о HTTP — принимают и возвращают доменные объекты
- Generic-репозитории (`BaseRepository`, `OwnedBaseRepository`) устраняют дублирование CRUD-операций
- Dependency Injection через FastAPI `Depends()` — композиция зависимостей по цепочке

**Что можно улучшить:**
- `HabitService` занимает 969 строк — стоит разбить на `HabitScheduleService` (логика расписаний) и `HabitTrackingService` (трекинг выполнений). Это улучшит тестируемость и читаемость.

### 2.2 Async-first

Весь стек работает асинхронно:
- FastAPI + Uvicorn (ASGI)
- SQLAlchemy 2.0 async с asyncpg
- Redis через aioredis-совместимый клиент
- httpx.AsyncClient для внешних запросов (Google OAuth, ZenQuotes)
- APScheduler для фоновых задач

Блокирующих вызовов не обнаружено — это правильно для I/O-bound приложения.

### 2.3 Модели данных

8 таблиц, все с UUID первичными ключами, UTC-таймстампами, каскадными удалениями:

| Модель | Назначение | Ключевые особенности |
|--------|-----------|---------------------|
| User | Пользователи | case-insensitive email (index на LOWER), nullable password для OAuth |
| OAuthAccount | OAuth-аккаунты | Composite unique (provider + user_id) |
| Theme | Темы-категории | Owner-scoped, unique(owner_id + name) |
| Task | Задачи | FK на Theme (SET NULL), FK на Priority |
| Priority | Приоритеты | Справочник с weight для сортировки |
| Habit | Привычки | JSON schedule_config, CHECK на schedule_type |
| HabitCompletion | Выполнения привычек | Unique(habit_id + date), CASCADE от Habit |
| Quote/QuoteBatch | Мотивационные цитаты | Batch-загрузка из внешнего API |

**Сильные стороны:**
- Правильные `ondelete` стратегии (CASCADE для владельца, SET NULL для опциональных связей)
- CHECK-ограничения на формат цвета
- Индексы на часто фильтруемых полях (schedule_type, ends_on, status)
- `OwnedModel` mixin обеспечивает изоляцию данных пользователей

## 3. Безопасность

### 3.1 Что сделано хорошо

**Хеширование паролей — Argon2:**
Используется `argon2-cffi` — победитель конкурса Password Hashing Competition. Это лучший выбор на 2026 год. Реализация корректна: `PasswordHasher.hash()` + `verify()` с обработкой `VerificationError` и `InvalidHashError`.

**CSRF-защита:**
Кастомная реализация с `secrets.token_urlsafe(32)` и `secrets.compare_digest()` (constant-time сравнение, защита от timing-атак). Токен проверяется из заголовков (`X-CSRFToken`) или формы.

**SQL-инъекции:**
Риск отсутствует. Используется исключительно SQLAlchemy ORM с параметризованными запросами. Единственное использование `text()` — в миграциях Alembic с `.bindparams()`.

**XSS:**
Jinja2 auto-escaping включён по умолчанию. Поиск по всему проекту не выявил использования `|safe`, `markup()` или отключения `autoescape`.

**Rate Limiting:**
Продуманные лимиты на аутентификацию:
- Login: 20 req/min по IP, 5 req/min по аккаунту
- Register: 10 req/min по IP, 3 req/hour по аккаунту
- OAuth: 30-60 req/min по IP

**Авторизация:**
Owner-scoped репозитории фильтруют все запросы по `owner_id`. Если пользователь не аутентифицирован, `_owner_filter()` возвращает `false()` — пустой результат гарантирован.

**Валидация ввода:**
Pydantic v2 схемы с `EmailStr`, ограничениями длины (пароль 8-256, название 46 символов), нормализацией email.

### 3.2 Что стоит усилить

**1. Security Headers (приоритет: СРЕДНИЙ)**

Отсутствуют HTTP-заголовки безопасности. Рекомендуется добавить middleware:

```python
# Пример middleware для security headers
@app.middleware("http")
async def add_security_headers(request, call_next):
    response = await call_next(request)
    response.headers["X-Content-Type-Options"] = "nosniff"
    response.headers["X-Frame-Options"] = "DENY"
    response.headers["X-XSS-Protection"] = "1; mode=block"
    response.headers["Referrer-Policy"] = "strict-origin-when-cross-origin"
    response.headers["Permissions-Policy"] = "camera=(), microphone=()"
    # CSP — настроить под используемые CDN (Font Awesome, Google Fonts)
    return response
```

Эти заголовки — стандартная практика, и их наличие покажет преподавателю осведомлённость о веб-безопасности.

**2. Cookie Security (приоритет: НИЗКИЙ для dev)**

Текущие настройки корректны для разработки:
- `SameSite=lax` — защита от CSRF через cookie
- `HTTPS_ONLY=False` — ожидаемо для localhost

Для продакшена нужно: `SameSite=strict`, `HTTPS_ONLY=True`, `Secure=True`.

**3. Rate Limiting на остальные эндпоинты (приоритет: НИЗКИЙ)**

Сейчас rate limiting только на auth. Для полноты можно добавить общий лимит (например, 100 req/min на пользователя) для защиты от abuse.

## 4. Производительность

### 4.1 N+1 проблема в привычках (приоритет: ВЫСОКИЙ)

В `HabitService.list_habits()` для каждой привычки выполняется отдельный запрос на получение дат выполнений:

```python
# Текущий код — N+1
for habit in habits:
    completion_dates = await self.habit_repo.list_completion_dates(habit.id)
```

При 20 привычках это 20 дополнительных запросов. Решение — batch-загрузка:

```python
# Рекомендация — один запрос на все привычки
habit_ids = [h.id for h in habits]
all_completions = await self.habit_repo.list_completion_dates_batch(habit_ids)
# Группировка по habit_id в Python
```

### 4.2 Статистика в памяти (приоритет: СРЕДНИЙ)

`StatisticsService` загружает все задачи и привычки пользователя в память и вычисляет метрики в Python. При малом объёме данных (до нескольких сотен записей) это приемлемо, но не масштабируется.

Альтернатива для будущего — агрегация на стороне БД:

```sql
SELECT
    COUNT(*) FILTER (WHERE completed_at IS NOT NULL) as completed,
    COUNT(*) as total
FROM tasks
WHERE owner_id = :uid AND created_at >= :since;
```

### 4.3 Connection Pool (приоритет: НИЗКИЙ)

Используются параметры пула по умолчанию из SQLAlchemy. Для дипломной работы это нормально, но в продакшене стоит настроить:

```python
engine = create_async_engine(
    url,
    pool_size=10,        # макс. одновременных соединений
    max_overflow=5,      # дополнительные при пиковой нагрузке
    pool_timeout=30,     # таймаут ожидания свободного соединения
    pool_recycle=3600,   # переподключение раз в час
)
```

### 4.4 Кеширование тем (приоритет: НИЗКИЙ)

Темы кешируются в dict на уровне одного запроса (`_get_theme_cached`). Для более агрессивного кеширования можно использовать Redis-кеш с TTL, но для текущего масштаба это избыточно.

### 5.2 Что можно добавить

**1. Нагрузочное тестирование (опционально для защиты):**
Простой скрипт с `locust` или `httpx` для демонстрации, что приложение выдерживает N одновременных пользователей.

**2. Property-based тесты (для бонусных баллов):**
Библиотека `hypothesis` для тестирования граничных случаев в логике расписаний привычек — это произведёт впечатление на преподавателя.

**3. Тесты безопасности:**
Проверка, что один пользователь не может получить доступ к данным другого — такие тесты уже частично есть в integration, но стоит выделить отдельно.

## 6. DevOps и CI/CD

### 6.1 Что есть — и это впечатляет

7-этапный CI pipeline в GitHub Actions:

```
pre-commit (lint/format/mypy)
    ↓
tests (unit + integration + coverage)
    ↓
docker-build (multi-stage image)
    ↓
docker-smoke (healthchecks в контейнере)
    ↓
migration-smoke (alembic upgrade + downgrade)
    ↓
security-python (pip-audit)
    ↓
security-container (Trivy scan)
```

Плюс отдельные workflows: CodeQL (SAST) и dependency-review.

### 6.2 Docker

Multi-stage сборка (builder → runtime), non-root пользователь, HEALTHCHECK в Dockerfile, docker-compose с health conditions. Это production-grade конфигурация.

### 6.3 Что можно добавить

**Structured logging (приоритет: СРЕДНИЙ):**

Текущее логирование — базовый `logging.basicConfig()`. Для продакшена стоит перейти на JSON-формат:

```python
import structlog

structlog.configure(
    processors=[
        structlog.stdlib.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ],
)
```

Это позволит использовать ELK/Grafana для агрегации логов.

## 7. Вопросы, которые может задать преподаватель

### 7.1 «Почему FastAPI, а не Django/Flask?»

Аргументы в пользу FastAPI:
- Нативная поддержка async/await — важно для I/O-bound приложения с БД и Redis
- Автоматическая валидация через Pydantic
- Dependency Injection из коробки
- Автогенерация OpenAPI/Swagger документации
- Высокая производительность (на уровне Go/Node.js для I/O задач)

### 7.2 «Почему UUID, а не автоинкремент?»

- Безопасность: автоинкремент позволяет перебирать ID (IDOR-атака)
- Распределённость: UUID генерируется на клиенте, не нужен round-trip к БД
- Миграция: при слиянии баз данных нет конфликтов ID

### 7.3 «Почему Argon2, а не bcrypt?»

- Argon2 — победитель Password Hashing Competition (2015)
- Устойчив к GPU-атакам (memory-hard алгоритм)
- Настраиваемые параметры: память, итерации, параллелизм
- Bcrypt ограничен 72 байтами пароля; Argon2 — нет

### 7.4 «Как защищён проект от SQL-инъекций?»

- Используется SQLAlchemy ORM — все запросы параметризованы
- Нет raw SQL в бизнес-логике
- Pydantic-схемы валидируют и типизируют входные данные до попадания в запрос

### 7.5 «Как обеспечена изоляция данных пользователей?»

- `OwnedModel` mixin добавляет `owner_id` ко всем пользовательским сущностям
- `OwnedBaseRepository` автоматически фильтрует все запросы по `owner_id`
- Если пользователь не аутентифицирован, фильтр возвращает пустой результат
- Это делает утечку данных между пользователями невозможной на уровне запросов

### 7.6 «Почему Redis для сессий?»

- Быстрый доступ (in-memory) — не нагружает PostgreSQL
- TTL-истечение сессий — автоматическая очистка
- Atomic-операции через pipelines — консистентность
- Горизонтальное масштабирование: сессии не привязаны к конкретному серверу

### 7.7 «Какие есть узкие места в производительности?»

Честный ответ — N+1 в загрузке привычек и вычисление статистики в памяти (см. раздел 4). Студент должен показать, что он осознаёт проблему и знает решение, даже если не реализовал его.

## 10. Заключение

HabitFlow — это проект, который демонстрирует **зрелый инженерный подход**:

1. **Архитектура** — чёткое разделение на слои, dependency injection, generic-паттерны
2. **Безопасность** — Argon2, CSRF, rate limiting, owner-scoping, параметризованные запросы
3. **Качество** — строгий линтинг (Ruff + mypy strict), pre-commit hooks, 80%+ test coverage
4. **DevOps** — multi-stage Docker, 7-этапный CI, security scanning, health checks
5. **Async** — полностью асинхронный стек без блокирующих операций

Основные точки роста: оптимизация N+1 запросов, security headers, structured logging. Эти улучшения не критичны для работающего приложения, но их упоминание на защите покажет, что автор понимает разницу между «работает» и «готово к продакшену».

Проект готов к защите. Главное — уметь объяснить принятые решения и знать, что можно улучшить дальше.
